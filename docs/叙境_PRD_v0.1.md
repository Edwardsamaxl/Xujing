🥷 叙境不是传统导览，也不是简单小游戏。它是一个用 AI 生成千人千面游览故事，并据此引导游客行为的景区叙事导流引擎。

# 叙境 x 故宫博物院：AI 叙事导流引擎 PRD v0.1

日期：2026-05-22
版本：v0.1
产品阶段：MVP / 故宫博物院试点验证
目标读者：创始团队、产品、技术、故宫博物院合作方、潜在投资人

## 1. 一句话定位

**叙境是故宫博物院 AI 叙事导流引擎：通过生成式讲解和自适应剧情路线，把游客兴趣转化为可调度的游览行为，帮助故宫缓解钟表馆与珍宝馆等热门专馆的拥堵、激活延禧宫与寿康宫等冷门区域。**

更短的对外表达：

> 我们用 AI 为故宫博物院的游客生成千人千面的游览故事，并根据现场客流动态改写剧情路线，让游客主动探索钟表馆、武英殿、延禧宫等冷门殿宇，帮故宫完成分流和运营复盘。

## 2. 背景与问题

故宫博物院年接待量超过 1700 万人次，是中国最受欢迎的博物馆。与有固定动线的场馆不同，故宫没有强制游览路线，游客自由扎堆在钟表馆、珍宝馆等热门专题展馆与中轴线区域，而延禧宫、寿康宫等偏离主游线的殿宇常年冷清，大量优质展品与历史场景被错过。

故宫要的不是多一个 AI 讲解，而是解决具体问题：
- 钟表馆、珍宝馆等热门专馆常年排队，如何自然分流？
- 延禧宫、寿康宫、慈宁宫等支线殿宇游客稀少，如何低成本激活？
- 游客的真实兴趣反馈（如"钟表馆排队太久"）如何变成运营动作？

机会在于把 AI 从"回答问题的导游"升级为"叙事驱动的运营工具"，用故事而不是命令引导游客完成故宫需要的行为。

## 3. 产品判断

叙境不是这些产品：

- 故宫语音导览 H5
- 智慧博物馆大屏
- 文旅小游戏
- 游客问卷系统
- 博物馆数据中台

叙境的定位只有一个：

**AI Narrative Orchestration for the Palace Museum / 故宫博物院 AI 叙事导流引擎**

核心能力不是"讲故事"，而是：

> **用故事驱动游客在故宫博物院内的行为。**

## 4. 目标用户与买方

| 角色 | 核心诉求 | 产品价值 |
|---|---|---|
| 游客 | 少排队、听懂故宫文物、获得探索感和奖励 | 千人千面的剧情导览，像解锁任务而不是被动听讲 |
| 故宫运营部 | 分流太和殿/乾清宫、激活钟表馆/武英殿/延禧宫、活动复盘 | 通过后台配置拥堵点和目标点，动态调整剧情路线 |
| 故宫管理层 | 观众服务质量、数据化运营 | 用试点数据证明分流效果和游客参与度提升 |

## 5. 游客、故宫、商户各得到什么

### 5.1 对游客

- 不只是听固定讲解，而是获得适合自己兴趣的故宫故事（如历史爱好者听到太和殿的皇家礼制，亲子家庭听到脊兽的"守护精灵"故事）。
- 不只是被动排队，而是在剧情中解锁新的殿宇（如被引导至钟表馆探索"停走的指针"）。
- 不只是逛完离开，而是获得故宫专属插画卡片或隐藏剧情。

### 5.2 对故宫博物院

- 把游客从钟表馆、珍宝馆等热门专馆自然引向延禧宫、寿康宫、慈宁宫等支线区域。
- 用低成本方式激活冷门展品（如延禧宫灵沼轩遗址、寿康宫黄花梨大柜、慈宁宫雕塑）和冷门殿宇。
- 通过游客行为和语义反馈生成运营复盘。
- 为后续智慧博物馆系统接入提供观众行为数据基础。

### 5.3 对商户（v0.2 规划）

MVP 阶段不涉及优惠券和商户核销。后续版本可扩展：
- 故宫文创店获得更自然的到店理由（"你找到了线索，文创店有对应的纪念章"）。
- 通过游客兴趣判断故宫文创方向（如历史人物系列 vs 神兽萌宠系列）。

## 6. 三层能力，从内容到行为

| 层级 | 功能 | 游客感受 | 故宫价值 |
|---|---|---|---|
| Generative 生成性 | 千人千面的 AI 讲解 | 我听到的是适合我的故宫故事 | 钟表馆、武英殿的冷门展品也能低成本生成吸引人的内容 |
| Adaptive 自适应 | 根据拥堵和兴趣动态改路线 | 我像在解锁剧情，不像被疏散 | 太和殿/乾清宫分流，钟表馆/武英殿/延禧宫引流 |
| Evidence 实证化 | 沉淀兴趣、路线、提问、到访数据 | 不需要额外填问卷 | 获得经营复盘和内容决策依据 |

## 7. MVP 只做到这一步

### 7.1 这两周必须做完（故宫博物院试点）

- 扫码即用的 H5 移动端页面，无需下载 App，手机浏览器直接打开。
- 游客选择兴趣标签：历史、建筑、人物、亲子、悬疑、工艺。
- **新增"秘辛地图"（Explore 页面）**：展示全部 6 个点位，每个点位以"秘辛预告"作为钩子，已勘验的地点盖"已阅"印章。
- 游客自主选择目标地点，或接受系统推荐的下一个地点；Explore 页提供 `[随机探索]` 按钮。
- 基于固定语料库 + DeepSeek API 生成个性化讲解（围绕太和殿脊兽、钟表馆机械钟、延禧宫灵沼轩等）。
- Narrative 页面只展示与游客兴趣标签匹配的单一叙事，不设多分支选项卡。
- 后台手动标记点位状态：正常、拥挤、暂停引导（MVP 通过 seed 数据配置，变更需技术侧调整）。
- 游客在 Explore 页面选择或随机一个目标地点。
- 系统实时判断游客定位，给出到达该地点的路线引导（Navigate）。
- 游客在 Navigate 页面点击"我已到达"：后台立即标记该地点为"已勘验"并盖章，同时弹出奖励小卡片（故宫专属插画/徽章），并自动解锁该地点的完整秘辛。
- 到达后直接进入 Narrative 剧情（基于兴趣标签的单一叙事）。
- **取消独立的 Reward 页面**；完整秘辛统一收纳在顶部导航栏"密档"入口中，游客可随时查看已解锁秘辛。
- 后台输出基础运营复盘：扫码、兴趣、路线、任务、反馈关键词。

### 7.2 试点范围（故宫博物院）

**本期覆盖 6 个核心点位（3 个出发点 + 3 个目的地），1 条主剧情线：**

> 策略：只挑选 3 个流量高压区作为"出发点"，3 个流量低谷区作为"引导目的地"。一共只需要维护 9 组碎片信息（即 3×3 的连线组合）。

**出发点（高压区）：**
- **钟表馆**（奉先殿，热门收费专馆，机械钟表专题，需分流）
- **珍宝馆**（宁寿宫区，热门收费专馆，宫廷珍宝专题，需分流）
- **武英殿·陶瓷馆**（"瓷母"镇馆之宝，特展期人流极高，需分流）

**目的地（低谷区）：**
- **延禧宫**（东六宫深处，灵沼轩遗址，外国文物/特展，需引流）
- **寿康宫**（外西路，崇庆皇太后原状陈列，需引流）
- **慈宁宫**（外西路，孝庄太后居所，雕塑馆，需引流）

### 7.3 这几件事暂时砍了

- 不做完整故宫智慧大屏。
- 不做安防级实时调度系统。
- 不强依赖故宫实时客流开放 API。
- 不主卖 IP 防盗、数字水印和版权维权闭环。
- 不做重 AR / 元宇宙效果。
- 不承诺替代人工安保疏散，产品定位是软性引导和运营增强。

## 9. 游客从扫码到离开故宫的完整路径

### 9.1 新版交互流程

```
Start（扫码进入）→ Interest（兴趣选择）→ Explore（秘辛地图）
  → Navigate（路线引导 + 到达打卡 + 奖励弹卡）→ Narrative（单点剧情）
  → [去下一个地点] → Navigate → ...
```

**Step 1 · 扫码进入（Start）**
- 游客在任意位置扫码，手机浏览器直接打开 H5 页面。
- 首屏极简，一句话欢迎 + "开始探索"按钮。

**Step 2 · 兴趣选择（Interest）**
- 游客选择兴趣标签：历史、建筑、人物、亲子、悬疑、工艺。
- 选择后该标签即固定，后续所有 Narrative 页面均基于该标签渲染单一叙事，不设多分支切换。

**Step 3 · 秘辛地图（Explore）—— 选关中枢**
- 页面展示全部 6 个点位（3 热门 + 3 冷门），以卡片网格形式呈现。
- 每张卡片显示：地点名称、一句"秘辛预告"（引流钩子）、当前状态标签。
- **状态标签**：
  - `未探索`：灰色边框，可点击。
  - `探索中`：金色边框，表示游客已选择但尚未打卡。
  - `已勘验`：红色印章覆盖（如"已阅""勘毕"），不可重复打卡。
- 顶部/底部提供 **`[随机探索]`** 按钮，方便游客快速开始，系统随机抽取一个未探索地点作为目标。
- **软引流机制**：系统根据当前拥堵状态，在地图顶部以弱提示形式推荐一个冷门点（如"寿康宫此刻人少，暗格中的佛珠正在等你"），但不强制，游客可忽略并自主选择任何地点。
- 点击未探索卡片（或随机）→ 进入 Navigate 页面。

**Step 4 · 路线引导、到达打卡与奖励（Navigate）**
- 系统根据游客当前 GPS 定位，计算到目标地点的步行路线、预计时间、距离。
- 若游客已在目标地点附近（如 GPS 距离 < 100m 或处于该殿宇地理围栏内），页面提示"你似乎已在此地附近"，`[我已到达]`按钮直接可用。
- 若目标地点为冷门点，文案包装为剧情引导而非管理指令（如"灵沼轩的地下图纸需要你去拼凑"而非"请前往延禧宫"）。
- 底部操作区：
  - **`[我已到达]`**（主按钮）：到达后点击，**三件事同时发生**：
    1. 后台标记该地点为`已勘验`，Explore 页面自动盖章；
    2. 弹出奖励小卡片（故宫专属插画/徽章），停留 2-3 秒后自动收起；
    3. 自动解锁该地点的完整秘辛（收纳在顶部导航栏"密档"入口中）。
    随后自动进入 Narrative 页面解锁剧情。
  - **`[更换目标]`**（次要按钮）：放弃当前目标，返回 Explore 页面自主选择其他地点。
- **设计意图**：到达即奖励， Explore 盖章 + 弹卡 + 解锁秘辛三层反馈在一个动作里完成，不跳转独立 Reward 页面。

**Step 5 · 剧情叙事（Narrative）**
- 到达后解锁该地点基于游客兴趣标签的单一叙事（title + baseContent + flavorText）。
- **不设"调查工艺细节 / 审问 NPC / 寻找线索"等多选项卡**，叙事内容一次性完整呈现，游客被动接收但可上下滑动阅读。
- 顶部导航栏常驻 **`密档`** 按钮，游客可随时点击查看已解锁的秘辛收藏册（全部 6 张秘辛默认上锁，到达后自动解锁对应卡片）。
- 页面按 5:2:3 分为三个视觉区域：
  - **顶部叙事区（~50%）**：深色半透明卡片，文本以打字机动效匀速呈现；顶部显示当前兴趣标签（如"历史 视角"）。
  - **中部思考反馈区（~20%）**：未交互时展示浅色引导文案，如"你可以试着问我：这扇窗为何半掩？"。
  - **底部语音交互区（~30%）**：
    - 麦克风按钮周围有缓慢跳动的金色呼吸光环，提示"可以对话"。
    - 按钮上方悬浮微小对话气泡，每 5 秒轮换一次提示文案（如"问我关于这里的秘辛…""问我关于建筑的工艺…"）。
    - **长按麦克风**：按钮变为红色波纹涟漪动效，中部文案变为"正在聆听您的疑问…"；松手后进入识别与回答流程。
    - **回答展示**：识别出的问题与 AI 回复以"金色引用框"形式插入叙事区下方，不覆盖原有叙事内容；AI 回复控制在 80 字以内。
- 底部操作区：
  - **`[去下一个地点]`**（主按钮）：系统根据当前位置、拥堵状态和游客兴趣，**推荐**下一个地点并进入 Navigate。
  - **`[查看其它视角]`**（次要按钮，视觉弱化）：展开同地点不同兴趣标签的叙事卡片（如当前为"历史"，可切换查看"悬疑版""人物版"等），增加停留与探索感。
- **关键点**：Narrative 专注当前地点内容消费。推荐是软性建议，不是强制命令。

**Step 6 · 循环或结束**
- 游客可重复 Step 3-5，直到探索完全部 6 个地点或主动结束。
- **出园结算页面**：设计待补充（见 §9.2）。

### 9.2 出园结算页（待设计）

游客点击"结束游览"或检测到长时间无操作后，需要一页总结性的结算界面。当前尚未确定最终方案，候选方向：

1. **秘辛收集册**：展示本次解锁的全部秘辛卡片，像一本翻阅的密档册。
2. **路线足迹图**：可视化本次游览的轨迹（去过哪些点、走了多少米、停留时长）。
3. **数字勋章墙**：根据到访点数、解锁秘辛数、探索完整度发放不同等级勋章。
4. **个性化结语**：AI 根据游客的兴趣标签和实际路线，生成一段专属总结文案。

> **决策待确认**：结算页的核心目标是"分享裂变"还是"情感闭合"？前者侧重生成可发朋友圈的纪念图，后者侧重给游客一个"今天没白来"的心理满足。

## 10. 故宫运营人员在后台的操作路径

1. 准备故宫资料：殿宇介绍（太和殿、乾清宫、钟表馆等）、历史故事、人物关系、故宫建筑信息。

3. 配置运营目标：钟表馆/珍宝馆/陶瓷馆分流、延禧宫/寿康宫/慈宁宫引流、亲子路线等。

## 12. AI 怎么生成故事而不胡编

### 12.1 固定语料库模式（MVP）

MVP 不采用 RAG，而是使用**人工编写的固定语料库 + AI 风格化润色**的双层架构：

- **语料库**：由运营团队预先编写，按 `点位 × 兴趣标签` 组织，存放于 `narrative_templates` 数据表。
- **AI 职责**：仅做语气适配、衔接润色、悬念设置，**不创造语料库之外的新事实**。

语料库三层字段：

| 字段 | 必填 | 作用 | 字数建议 |
|---|---|---|---|
| `title` | 是 | 该点位对该兴趣标签的叙事标题 | 10-20 字 |
| `baseContent` | 是 | 核心讲解内容，所有事实依据 | 150-300 字 |
| `flavorText` | 否 | 氛围/风格提示，辅助 AI 调整语气 | 20-50 字 |

内容安全红线：
- 重要历史事实必须能在故宫官方资料中找到出处。
- 低置信度内容不得编造成确定事实（如"据传""可能"等模糊表达不得被 AI 强化为肯定语气）。
- 涉及文物年代、人物关系、历史事件时必须保守表达。

### 12.3 同一件展品，AI 能讲出不同版本

以**钟表馆铜镀金写字人钟**为例：

- 历史版：铜镀金写字人钟由英国伦敦威廉森制造，通高 2.31 米，是 18 世纪清宫收藏的西洋贡钟代表作。底层设有独立的写字机械人，上弦后单腿跪地，执笔写下"八方向化，九土来王"八个汉字。这八个字出自《诗经》，原意是万方来朝，却被安放在一个西洋机械人手中，成为中西政治象征的奇特混合体。
- 亲子版：这个钟表里藏着一个小人！上弦之后，他会单腿跪下来，用毛笔写出"八方向化，九土来王"八个字。你知道他为什么要跪着写吗？因为他是从英国来的"客人"，在皇帝面前必须行礼。他写完字还会摇头晃脑，好像在说"我写完了！"
- 悬疑版：写字人钟底座的纹路并非欧洲常见纹样，而是与延禧宫灵沼轩（水晶宫）的钢铁构件出自同一批进口物资。更蹊跷的是，钟内机芯刻有 1773 年的编号，但故宫档案记载它直到 1793 年才由马戛尔尼使团带入清宫。这中间的 20 年，这座钟究竟在哪里？
- 人物版：写字人钟写下"八方向化，九土来王"时，头部会随之摆动。据说乾隆第一次看到它写字时，认为这个西洋小人是在向自己"磕头称臣"，龙颜大悦。但英国制造商的原始说明书上写着，机械人低头是为了调整笔尖角度——一个技术动作，被解读为政治臣服。
- 建筑工艺版：写字人钟内部有两套独立机械系统：一套控制走时和报时，另一套专门驱动写字机械人。两套系统共用一根发条轴，但通过差速齿轮实现动力分配。这种"一芯双驱"结构，在 18 世纪欧洲钟表中也极为罕见。
- 工艺版：机械人的手指由铜合金制成，表面覆盖一层极薄的鹿皮以模拟人手握笔的摩擦力。笔尖角度经过精密计算，与纸面保持 45 度，这个角度既能保证墨迹均匀，又不会划破宣纸——欧洲机械师为了适应中国毛笔，专门调整了书写机构。

以**寿康宫崇庆皇太后原状陈列**为例：

- 历史版：寿康宫是乾隆为生母崇庆皇太后建造的颐养之所，2015 年首次开放原状陈列。后殿东梢间复原了太后日常起居的通体大炕，西次间陈设着故宫现存最大的黄花梨雕龙大柜。2015 年修缮时，工作人员在大柜暗格中发现了上百件珍宝，包括红珊瑚如意、金錾花嵌珠杯盘等。
- 亲子版：寿康宫里有一个超级大的木头柜子，比两个大人加起来还高！2015 年，叔叔阿姨们修这个柜子的时候，在里面的秘密小格子里发现了好多宝贝，有红色的珊瑚如意、金色的珠珠杯子……这些宝贝是谁藏的呢？就是住在这里的老佛爷！
- 悬疑版：后殿西次间的黄花梨大柜暗格中，发现了红珊瑚如意、金錾花嵌珠杯盘等上百件珍宝。但奇怪的是，暗格的锁扣是朝内安装的——这意味着只能从柜子内部打开。难道有人住在柜子里？还是这些珍宝是在某种紧急情况下，从里面被藏进去的？
- 人物版：崇庆皇太后八旬万寿时，乾隆命苏州工匠特制紫檀攒竹缂丝嵌玉屏风宝座，作为寿礼。屏风上的"海屋添筹"图案，寓意长生不老。但鲜为人知的是，缂丝工艺极其耗时，这件屏风用了整整三年才完成——而那时太后已经 77 岁了，乾隆是在为母亲的"下一个十年"提前做准备。
- 建筑工艺版：寿康宫后殿的"通体大炕"并非砖砌，而是以黄土、白灰、沙子按比例混合夯制而成，表面再铺设金砖。这种炕体蓄热性能好，散热均匀，是北京地区满族民居与宫廷建筑结合的典范。
- 工艺版：紫檀攒竹缂丝嵌玉屏风宝座采用"攒竹"工艺——将竹筒破成细丝，按图案编织成底，再在上面缂丝。缂丝"通经断纬"，每一根纬线只在需要的地方出现，形成图案边界如刀刻般清晰，故有"一寸缂丝一寸金"之说。

### 12.4 自适应策略（软性推荐）

系统根据以下因素**推荐**下一目标地点，但游客始终拥有完全自主权：

- 游客兴趣
- 当前展厅拥堵状态
- 故宫希望引导的目标展厅（冷门点优先）
- 展厅间步行距离
- 商户权益
- 游客已完成任务（已勘验地点不再推荐）
- 活动时间段

**推荐呈现方式**：

1. **Explore 页面顶部弱提示**：以一行轻量文案形式出现，不弹窗、不强制。如"寿康宫此刻人少，暗格中的佛珠正在等你"，旁边附带 `[去看看]` 和 `[忽略]` 两个按钮。
2. **Narrative 页面底部推荐**：游客点击"去下一个地点"时，系统给出推荐目标及理由，游客可选择 `[接受]`、`[换一个]`（重新随机）或 `[回地图自选]`。

**示例**：

> 如果钟表馆拥挤，且游客选择了"悬疑故事"，系统不提示"请不要前往钟表馆"，而是在 Explore 顶部弱提示："写字人钟的底座纹路还缺半块拓片，完整图纸藏在延禧宫的灵沼轩残图里。那里现在人很少，可以先去拼凑出那段中西合璧的工法秘密。" 游客可忽略并自主选择其他地点。

> 如果亲子游客在珍宝馆完成 Narrative 后点击"去下一个地点"，系统推荐："这些金银细工缺少温度，寿康宫里藏着老佛爷手制的珐琅小件，找到它们可以集齐一套宫廷印记。" 游客可点击 `[接受推荐]`、`[重新随机]` 或 `[回地图自选]`。

### 12.5 反馈总结

系统对游客提问和反馈做主题归类：

- 内容太浅 / 太深
- 展厅灯光 / 动线问题
- 钟表馆 / 珍宝馆排队问题
- 故宫文创价格问题
- 故宫餐厅服务问题
- 亲子友好度
- 故宫文创偏好（历史人物 / 神兽萌宠 / 建筑纹样）

默认只输出匿名聚合结果，不展示个人可识别信息。

### 12.6 语音交互技术链路（P1，可选）

Narrative 页面的长按麦克风提问并非演示动效，而是一条完整的语音→文字→AI→展示链路。由于涉及额外 ASR 服务接入，列为 P1，可根据试点时间酌情实现。

**完整链路**

```
用户长按麦克风
  → 前端 Web MediaRecorder API 录制音频（webm/wav，上限 10 秒）
  → 松手后音频 blob POST 至后端 /api/voice/ask
  → 后端调 ASR API（Whisper / 讯飞 / 阿里云）转文字
  → 文字 + 当前 spotId + 游客 interestTag 注入 System Prompt
  → 调 DeepSeek API（deepseek-chat）生成回答（max_tokens 120）
  → 后端返回 { question: '识别文字', answer: 'AI 回答' }
  → 前端在金色引用框中展示问答，不覆盖原有叙事内容
```

**System Prompt 预注入（防胡编）**

```
你现在是故宫的数字秘辛讲述者。请始终保持神秘且克制的语气。
当前地点：{spotName}
游客兴趣标签：{interestTag}
以下是为该地点人工编写的权威语料，你的回答必须基于这些内容，
禁止编造语料库之外的历史事实：
---
title: {title}
baseContent: {baseContent}
flavorText: {flavorText}
---
无论游客询问什么，回复必须紧扣当前地点的地理背景和历史文化，
回复字数控制在 80 字以内，确保易读性。
```

| 限制项 | 说明 |
|---|---|
 HTTPS | H5 调用麦克风必须在 HTTPS 环境下运行，本地调试需配置证书或走内网穿透 |
 麦克风权限 | iOS Safari / Android Chrome 均会弹授权框，需在产品说明中告知游客 |
 户外嘈杂环境 | 故宫室外游客密集，ASR 识别率可能下降；建议单次录音上限 10 秒，过短可提示"请靠近手机重新提问" |
 成本 | Whisper API 约 $0.006/分钟；国内替代方案：讯飞语音听写、阿里云一句话识别 |
 最小可行方案 | 若时间不足，可先实现"长按→文字输入框"模式，跳过 ASR，直接让游客打字提问，LLM 链路不变 |

**TTS（文字转语音，可选增强）**

若希望 AI 真正"发声"而非仅展示文字，可在链路末端加入 TTS 服务（如 Azure Neural TTS、百度语音合成），将 DeepSeek 返回的文字转为音频流播放。此功能为体验增强项，不影响核心链路跑通。

## 15. 这几件事最可能出问题

| 风险 | 等级 | 应对 |
|---|---|---|
| 游客不按路线走（故宫殿宇间距离较远） | 高 | 把导流包装成剧情线索、奖励和减少排队，而不是管理命令；任务距离设为"步行 5 分钟内"的相邻殿宇 |
| 故宫不给实时客流数据 | 高 | MVP 先用人工标记拥堵和自有埋点，不把实时接口作为前置条件 |
| 故宫展品内容生成出错（历史敏感） | 高 | 固定语料库人工编写、敏感词过滤、模板化约束、人工审核镇馆之宝等重点展品 |
| 被认为只是故宫导览 H5 | 中 | 销售材料聚焦钟表馆/武英殿/延禧宫到访率、插画卡片领取率，而不是 AI 讲解 |
| 合规压力（博物馆观众数据） | 中 | 明确告知数据用途，只做必要采集，默认输出匿名聚合数据 |
| 手机浏览器兼容性问题 | 中 | 优先适配 iOS Safari 和 Android Chrome，避免使用过于前沿的 CSS/JS API |

## 18. 待确认问题（故宫博物院）

- 故宫是否允许在钟表馆、珍宝馆等位置放置扫码物料？
- 故宫买单方是谁：开放管理部、宣传教育部、文创部，还是院领导层？
- 是否已有可用的故宫殿宇资料、展品介绍和开放区域地图？
- 故宫首个试点的核心指标优先级是分流钟表馆/珍宝馆、激活延禧宫/寿康宫/慈宁宫，还是提升游客参与度？
- 团队是否有故宫渠道和现场执行能力？这会决定产品销售周期。
- 故宫重点展品（钟表馆藏品、珍宝馆藏品、寿康宫原状陈列等）的讲解内容是否需要故宫专家审核？

## 19. 结论

第一版不做完整智慧博物馆平台，只做 2 周内在故宫博物院能试点的 AI 叙事导流活动。

只要能证明故宫游客会被个性化剧情从钟表馆/珍宝馆引到延禧宫/寿康宫/慈宁宫，并留下到访记录，产品就值得继续系统化。

产品判断只有一条：

> **不要证明 AI 会讲故宫的故事，要证明 AI 生成的故宫故事能改变游客的展厅路线。**
